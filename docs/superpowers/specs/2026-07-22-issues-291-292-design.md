# Issues 291 and 292 Design

## Status

Approved design for GitHub issues #291 and #292. The two issues are implemented
on one branch and reviewed in one non-squash pull request, with one implementation
commit per issue after this design commit.

## Scope

This batch contains two ordered but otherwise independent changes:

1. #291 corrects the message action toolbar's Reply and Reply in thread
   presentation and routing.
2. #292 prevents timeline network side effects and background completions from
   blocking cached room navigation, while retaining durable convergence of the
   newest desired read state.

The GUI change lands first because it is local and provides a clean independent
commit. The Core scheduling change follows and owns the batch's architectural
risk. There is no shared product-state shortcut between them.

## Issue 291: Distinct Reply Actions

### Product Contract

For actionable rows in a room timeline, the inline action order is:

1. Add reaction
2. Reply
3. Reply in thread
4. Edit
5. Pin or unpin
6. More

Conditional actions may disappear, but the relative order of the remaining
controls does not change. Existing Rust-projected action eligibility remains
authoritative. React does not infer permissions or revive actions on redacted,
hidden, synthetic, or otherwise non-actionable rows.

Ordinary Reply uses the bent reply-arrow icon and calls
`onReply(roomId, eventId)`. Reply in thread uses the existing speech-bubble icon
and calls `onOpenThread(roomId, eventId)`. The thread action is offered only in
the main room presentation context. It is absent from thread and focused
timelines, so the UI cannot create a nested thread or expose a room-only action
from a focused projection.

Existing thread-summary chips are unchanged. They remain a separate entry point
and call the same `onOpenThread` handler for an existing root.

### Accessibility and Localization

English and Japanese catalogs retain the existing ordinary-reply label and add
a distinct timeline-scoped Reply in thread label. Both icon-only buttons use the
localized string as their accessible name and tooltip contract. Icons are
decorative and do not supply an alternative accessible name.

### Verification

`TimelineView` tests drive a room row and assert:

- exact action order with all controls available;
- stable relative order when Edit or another optional action is absent;
- the reply-arrow and speech-bubble are distinguishable by a stable test-facing
  icon marker rather than SVG path text;
- Reply invokes only `onReply`;
- Reply in thread invokes only `onOpenThread`;
- accessible names are distinct;
- Reply in thread is absent in thread and focused presentation contexts;
- redacted, hidden, and non-actionable rows gain no invalid reply action; and
- the existing thread-summary chip still opens the same root.

Catalog tests assert English and Japanese values. Existing reply composer,
thread-opening, typecheck, lint, and production build checks remain green.

## Issue 292: Navigation-Safe Read-State Side Effects

### Problem Boundary

Room selection and initial projection are already cache-fast. The failure is
downstream backpressure:

- `TimelineActor` directly awaits receipt and fully-read network requests;
- a blocked actor cannot acknowledge live-tail cancellation or drain later
  control messages;
- `TimelineManagerActor` waits without a deadline for that acknowledgement;
- background completion traffic and a full normal mailbox can delay a later
  subscription; and
- `AccountActor` can consequently remain inside timeline routing instead of
  processing the next foreground room-selection command.

The fix does not rewrite the actor hierarchy. It separates foreground
navigation, coalescible read-state admission, and background completion into
explicit lanes with owned state.

### Foreground Navigation Lane

`TimelineManagerHandle` gains a bounded foreground control lane for subscribe,
unsubscribe, and navigation-preemption messages. The manager loop checks this
lane before ordinary timeline commands and background completions. A committed
`SelectRoom` therefore cannot sit behind receipt completions, gap-repair
updates, link-preview results, or live-tail refresh results already queued on
the ordinary lane.

Foreground delivery remains reliable. It is never converted to a lossy
`try_send`. If the foreground lane is full, its bounded admission is surfaced as
a correlated failure instead of a silent no-op. Named capacities follow the
repository's large-account channel rules.

The reducer still commits Rust-owned navigation before the target subscription
performs optional network work. A cached target publishes its initial
projection without waiting for the previous room's side effects.

### Coalescing Read-State Inbox

Read-state commands do not wait for a timeline actor mailbox. The manager handle
owns a private shared coalescing inbox and a bounded wake channel:

- admission records the intent before returning;
- at most one wake is necessary, because a pending wake causes the manager to
  drain the complete inbox;
- a full wake channel does not lose data; the already-queued wake observes the
  newest inbox state; and
- the inbox is private Core state and never crosses the WebView boundary.

The inbox is keyed by account, room timeline target, and read-state kind:

- public room receipt;
- private/thread receipt; and
- the atomic fully-read plus private-receipt bundle used by Mark read.

Every accepted user command retains its `request_id`. Coalescing does not erase
terminal accounting. A successful newer target satisfies older waiters whose
targets it dominates. A command that times out or fails receives one correlated
failure, while its desired target may remain as an internal convergence intent;
later retry success updates authoritative state without emitting a second
terminal result for that request.

### Monotonic Desired State

The subscribed `TimelineActor` remains the authority for comparing visible
event positions because it owns canonical timeline order. Each actor publishes
an atomically replaced, read-only position index through its handle. The index
contains only actor generation, ordered event ranks, and the actor-known
confirmed read positions; it is not product state and is never serialized.
Canonical projection updates replace this index before later commands can use
the new order.

The manager normalizes a raw read command against that handle snapshot without
waiting for the actor mailbox. The resulting intent contains the event
identity, timeline kind, actor generation, and actor-known monotonic position
evidence. If the event is not in the current index, admission retains it as an
unordered pending target for later SDK/sync reconciliation rather than blocking
the command lane or guessing an order.

A newer desired position replaces an older desired position. An older viewport
observation, stale actor generation, stale completion, or delayed retry can
never replace a newer desired or confirmed position. When two candidates cannot
be ordered from the current canonical window, Core retains a small bounded
candidate set with their waiters until SDK receipt state or a later timeline
projection proves their order. It then keeps only the greatest target. It does
not guess from timestamps, event-id text, submission order, or completion
order.

Fully-read convergence treats the fully-read marker and private read receipt as
one desired operation. It never advances one locally while claiming the pair
succeeded if the SDK request failed.

### Owned Network Workers

The manager owns cancellable read-state workers in a task supervisor. It never
awaits SDK network I/O in its mailbox loop.

Each operation carries account/session generation, actor generation where
applicable, read-state kind, and operation generation. Completion re-enters the
manager as a tagged message. Only the current operation for the current session
may mutate confirmed state, emit success, or wake another operation.

There is at most one in-flight request per key and kind. When a dominating
target arrives, Core cancels or supersedes the older task and schedules only the
newest desired target. Every SDK wait has an absolute timeout. Timeout,
cancellation, disconnect, logout, account switch, and shutdown settle task
ownership; none leaves a detached worker or a retained mailbox owner.

Retry uses capped exponential backoff plus real external wakes from session
reconnect, room subscription checkpoints, and authoritative receipt updates.
Multiple viewport observations during an outage remain one desired target and
cannot create a request storm.

### Durable Restart Convergence

Unconfirmed desired read state is account-scoped private data. It is persisted
through `StoreActor` using the existing local unlock secret and a dedicated
HKDF domain, separate from composer drafts and scheduled sends. The payload is
versioned, size-bounded, written atomically, and contains only the minimum room,
timeline-kind, bounded event-target candidate set, and operation-kind data
needed to retry.

`AccountActor` owns persistence because it owns `StoreActor` and the active
session key. `TimelineManagerActor` owns runtime coalescing. They communicate
through a typed private persistence snapshot/wake path:

- login or restore loads the encrypted outbox before enabling retries;
- runtime desired/confirmed changes schedule a debounced save;
- confirmed entries are removed from the persisted payload;
- logout/account reset removes the account payload with the other local data;
- account switch fences late saves by session generation; and
- orderly shutdown performs a bounded final save after worker cancellation.

After restart, Core first compares a restored desired target with authoritative
SDK/sync receipt state. If sync proves the server is already at or beyond the
target, Core deletes the entry without sending. Otherwise it retries the newest
desired operation. A target that cannot yet be ordered remains pending until a
later projection or sync update supplies proof.

No persisted read-state data enters `AppState`, Tauri DTOs, TypeScript types,
logs, diagnostics, or QA stdout.

### Bounded Live-Tail Cancellation

The live-tail scheduler invalidates the operation generation before attempting
network cancellation. Sending `CancelLiveTailNetwork` and waiting for its
acknowledgement share one short absolute deadline. Timeout records a coarse
outcome and the manager continues foreground navigation. A late acknowledgement
or refresh completion is stale by operation generation and cannot publish a
snapshot or restart work.

This deadline is a safety bound, not a polling loop or correctness timer.

### Gap Repair and Other Background Work

Gap repair keeps its existing generation, publication, and render-ACK fences.
Link previews, pagination, and gap inspection continue to use owned tasks. Their
completion messages use the ordinary background lane and cannot overtake the
foreground navigation lane.

Room change ends foreground gap demand for the previous room immediately. It
does not wait for the previous repair task, projection ACK, or link-preview
completion. Late results remain fenced by actor and operation generation.

### Diagnostics and Privacy

Private-data-free diagnostics record only coarse tokens and counts:

- navigation admitted, committed, or failed;
- read intent admitted, coalesced, retried, timed out, or converged;
- retry source (backoff, reconnect, checkpoint, or sync reconciliation);
- stale completion rejected;
- live-tail cancellation acknowledged or timed out; and
- persisted outbox entry counts and load/save outcomes.

Diagnostics never contain room, event, transaction, account, or user IDs;
message content; pagination tokens; persisted payloads; or raw SDK errors.

## State and Ownership Invariants

1. Navigation reducer commitment and cached initial projection never await a
   timeline network side effect.
2. Every spawned task has one owner and a bounded cancellation/join path.
3. Every accepted foreground command receives exactly one correlated terminal
   outcome.
4. Read-state coalescing retains the newest provable target and never regresses
   confirmed state.
5. Wake coalescing may collapse notifications but never desired data.
6. Late session, actor, operation, or persistence generations cannot mutate the
   current account.
7. A success event is emitted only after SDK or sync confirmation.
8. Durable desired state is encrypted and never exposed to React.

## Test-First Verification

### Core Unit and Actor Tests

Tests first reproduce each failure with controlled never-resolving futures:

- stalled public receipt does not block a later cached subscription;
- stalled fully-read bundle does not block a later cached subscription;
- stalled live-tail refresh and missing cancellation ACK do not block a later
  subscription past the fixed navigation deadline;
- A -> B -> C foreground selections converge to C;
- a large background completion burst cannot starve the foreground lane;
- a newer read target cancels/coalesces an older one;
- stale completions cannot regress or emit success;
- public, private/thread, and fully-read operations remain distinct;
- retry after reconnect sends only the newest desired target;
- duplicate wakes do not create duplicate SDK requests;
- actor/session replacement settles all workers; and
- every submitted selection receives one bounded terminal outcome.

### Persistence Tests

- pending desired state round-trips through encrypted storage;
- plaintext room/event identifiers do not occur in the persisted file;
- corrupted or wrong-key payloads fail closed without exposing raw errors;
- confirmed state removes the durable entry;
- session replacement rejects a stale debounced save;
- restart retries a server-behind target;
- restart removes a target already satisfied by sync; and
- reset-local-data removes the payload.

### Regression Gates

- existing unread-count, public/private/thread receipt, fully-read, navigation,
  gap-repair, live-tail freshness, link-preview, and intent-lifecycle tests;
- a production-shaped large-account action/completion burst;
- `cargo test -p koushi-core --lib` and focused integration binaries;
- `cargo check --workspace --all-targets`;
- focused `TimelineView` and i18n tests, desktop typecheck, lint, and build; and
- local headless `live_signals` plus cached room-navigation scenarios with the
  network proxy disabled or stalled as required by each proof.

## Non-Goals

- Rewriting all actors into one global priority executor.
- Treating cancellation or timeout as receipt success.
- Dropping accepted read intents to make navigation appear fast.
- Deriving Matrix ordering or read semantics in React.
- Changing thread-summary behavior, composer semantics, or Rust action
  eligibility for #291.
- Solving unrelated dependency audit warnings.
