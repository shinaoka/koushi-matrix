# Internationalization And Locale Display Profile

Status: normative. This document defines the React-independent i18n substrate.
Dated plans may describe implementation order, but they must not contradict
this profile contract.

Last amended: 2026-06-15.

## Ownership

- Locale product state is `matrix_desktop_state::LocaleSettings`, persisted
  through the Rust-owned settings store.
- The app-facing locale/display profile is resolved in Rust by
  `matrix_desktop_state::resolve_locale_display_profile(settings, platform)`.
  React, browser fakes, and Tauri adapters consume the resolved profile; they
  must not parse raw language tags in feature components.
- GUI code may apply the resolved `lang`, `dir`, catalog locale, pseudo mode,
  and modifier labels to presentation. It must not make locale choice, text
  direction, pseudo-locale behavior, or platform shortcut labels a React-owned
  source of truth.

## Canonical Profile Shape

The serialized frontend shape is:

```json
{
  "lang": "en",
  "dir": "ltr",
  "catalog_locale": "en",
  "pseudo_locale": "none",
  "platform": "linux",
  "modifier_labels": {
    "primary": "Ctrl"
  }
}
```

Field meanings:

- `lang`: sanitized root language tag for the rendered product chrome. It is
  never the raw persisted input.
- `dir`: resolved layout direction, `ltr` or `rtl`.
- `catalog_locale`: message catalog selector, currently `en`, `ja`, or
  `pseudo`.
- `pseudo_locale`: pseudo-locale mode, `none`, `accented`, or `bidi`.
- `platform`: display platform profile, `macos`, `windows`, or `linux`.
- `modifier_labels.primary`: platform modifier label for shortcut display:
  `Cmd` on macOS, `Ctrl` on Windows/Linux.

## Resolution Rules

- Default settings resolve to English, LTR, non-pseudo catalog/profile.
- `en-*` resolves to English catalog and root `lang: "en"`.
- `ja-*` resolves to Japanese catalog and root `lang: "ja"`. The shipped
  Japanese catalog is complete for all `MessageId` values. English-identical
  values are allowed only through an explicit proper-noun/symbol allowlist
  covered by catalog tests.
- `en-XA` resolves to accented pseudo locale with LTR layout.
- `ar-XB` resolves to bidi pseudo locale with RTL layout.
- Recognized RTL language families such as Arabic, Hebrew, Persian, Urdu, and
  related tags resolve to RTL under `TextDirectionPreference::Auto`, even when
  the current product catalog falls back to English.
- Explicit `TextDirectionPreference::Ltr` or `Rtl` overrides script defaults.
- Invalid, unsupported, or private-use-like language input falls back to a
  sanitized English profile. Resolved profiles and diagnostics must not echo
  raw locale input.

## Catalog And Formatting Rules

- User-visible product text uses stable catalog IDs from
  `apps/desktop/src/i18n/messages.ts` or its successor.
- The React app activates the Rust-resolved catalog by consuming
  `AppState.locale_profile`; feature components do not choose catalogs or
  parse locale tags.
- Catalog templates use named placeholders such as `{roomName}`. Interpolation
  values are caller-owned data and must not be transformed by pseudo-localizer
  code.
- Pseudo-localization expands catalog-owned label text while preserving
  placeholders exactly. Tests must cover long labels, CJK samples, RTL samples,
  combining marks, and a bidi mode that is distinguishable from accented
  pseudo-localization.
- Core and adapters return structured codes, enum variants, and non-secret
  data. They do not return prose for product UI rendering.
- Remote/user text such as room names, display names, message bodies, snippets,
  attachment names, and reaction keys is rendered with `dir="auto"` at the UI
  boundary.
- CJK display and search policy is Rust-owned. Normalization, CJK-aware
  collation keys for room/person ordering, search matching policy, and verified
  highlight offsets are produced by core/search DTOs. React must not tokenize
  Japanese text, compute CJK sort keys, or reinterpret highlight ranges.
- CJK search normalization uses NFKC-style folding plus lowercase comparison for
  query and canonical text matching. It folds full-width/half-width Latin,
  numeric, and kana variants for exact-result verification while returning
  highlight offsets against the original visible text in UTF-16 code units.
- Core search may query the SDK index with raw and Rust-normalized query
  variants, but final results are accepted only after
  `matrix-desktop-search` verifies the original query against the canonical
  document text. SDK ngram candidates remain a candidate source, not the
  display source of truth.
- CJK room/person ordering uses Rust-produced display sort keys. The current
  policy is deterministic Japanese-oriented NFKC/lowercase folding with
  numeric segments ordered numerically. Locale-perfect dictionary ordering is
  deferred until an ICU-backed policy is approved; React still consumes only
  Rust-owned sort order or keys.
- CJK line-breaking and truncation are GUI presentation concerns. React/CSS may
  apply `line-break: strict`, `word-break: normal`, `hyphens: none`, logical
  spacing, wrapping, and ellipsis to Rust-owned text, but must not change the
  underlying room/message/member strings or locally sort/query/fold them.
- IME send-vs-commit semantics are part of the shared Rust composer key
  resolver. GUI code passes typed key facts, including `is_composing`, and never
  sends, inserts a product newline, accepts autocomplete, or closes a composer
  while Rust resolves the keypress as IME composition input.
- Layout CSS uses logical properties for locale-sensitive spacing and
  positioning. Physical left/right assumptions are allowed only when the
  physical direction is intentionally part of the visual design and documented
  in the component.

## Snapshot And Test Mirror Rules

- Tauri snapshots include `LocaleDisplayProfile` beside settings state. Adding
  or changing profile fields requires updating the hand-maintained Tauri DTO,
  TypeScript domain types, browser fake snapshots, Tauri IPC mocks, app
  harness snapshots, and DTO serialization-contract tests in the same change.
- Browser fakes and headless harnesses may mirror the Rust resolver only for
  deterministic tests. Product UI code must still consume the serialized
  profile rather than re-resolving it in React.
- Structured UI registries such as shortcut and menu definitions store catalog
  IDs for user-facing labels and notes. They are not a reason to put
  user-facing prose directly into feature components or domain registries.

## CJK Implementation Search Paths

When changing or reviewing Japanese/CJK behavior, inspect these paths before
coding:

- Catalog and pseudo-locale: `apps/desktop/src/i18n/messages.ts` and
  `apps/desktop/src/i18n/messages.test.ts`.
- Locale profile, CJK normalization, and display sort keys:
  `crates/matrix-desktop-state/src/locale_profile.rs` and
  `crates/matrix-desktop-state/tests/locale_display_profile.rs`.
- Search document verification, query variants, and highlight offsets:
  `crates/matrix-desktop-search/src/document.rs`,
  `crates/matrix-desktop-search/src/verify.rs`, and
  `crates/matrix-desktop-search/tests/search_adapter.rs`.
- Search actor candidate fan-out and final verification:
  `crates/matrix-desktop-core/src/search.rs`.
- GUI tests may assert rendered catalog output and DOM layout, but they must
  not add local tokenization, normalization, collation, or highlight repair.

## Headless Gates

Phase A gates:

- Rust unit tests for locale/display profile resolution, serialization,
  platform modifier labels, pseudo-locale tags, RTL/CJK handling, and
  private-data-free diagnostics.
- Catalog unit tests for ID coverage, pseudo expansion, placeholder
  preservation, RTL/CJK/combining samples, and interpolation behavior.
- Rust tests for Japanese catalog completeness accounting, CJK normalization,
  CJK collation key stability, exact search highlight offset preservation, and
  `is_composing` key handling across main, thread, edit, and autocomplete
  composer contexts.

Phase B gates:

- Browser-headless tests apply the Rust-resolved root `lang`/`dir`, render
  pseudo/CJK/RTL/long labels without clipping or overlap, verify
  `dir="auto"` on remote/user text, assert Japanese shell labels under
  `catalog_locale: "ja"`, and cover CJK text fitting on room names, sender
  names, message bodies, thread labels, and search snippets.
- A raw user-facing string scan or reviewed allowlist is part of the UI gate
  before #4 can be closed.
- CSS contract tests reject unreviewed physical `left`/`right`,
  `margin-left`/`margin-right`, `padding-left`/`padding-right`, and
  `border-left`/`border-right` declarations in locale-sensitive shell layout.
