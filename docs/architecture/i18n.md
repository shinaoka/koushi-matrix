# Internationalization And Locale Display Profile

Status: normative. This document defines the React-independent i18n substrate.
Dated plans may describe implementation order, but they must not contradict
this profile contract.

Last amended: 2026-06-14.

## Ownership

- Locale product state is `matrix_desktop_state::LocaleSettings`, persisted
  through the Rust-owned settings store.
- The app-facing locale/display profile is resolved in Rust by
  `matrix_desktop_state::resolve_locale_display_profile(settings, platform)`.
  React, browser fakes, and Tauri adapters consume the resolved profile; they
  must not parse raw language tags in feature components.
- GUI code may apply the resolved `lang`, `dir`, catalog locale, pseudo mode,
  and modifier labels to presentation. It must not make locale choice, text
  direction, pseudo-locale behavior, or platform shortcut labels a React-owned
  source of truth.

## Canonical Profile Shape

The serialized frontend shape is:

```json
{
  "lang": "en",
  "dir": "ltr",
  "catalog_locale": "en",
  "pseudo_locale": "none",
  "platform": "linux",
  "modifier_labels": {
    "primary": "Ctrl"
  }
}
```

Field meanings:

- `lang`: sanitized root language tag for the rendered product chrome. It is
  never the raw persisted input.
- `dir`: resolved layout direction, `ltr` or `rtl`.
- `catalog_locale`: message catalog selector, currently `en`, `ja`, or
  `pseudo`.
- `pseudo_locale`: pseudo-locale mode, `none`, `accented`, or `bidi`.
- `platform`: display platform profile, `macos`, `windows`, or `linux`.
- `modifier_labels.primary`: platform modifier label for shortcut display:
  `Cmd` on macOS, `Ctrl` on Windows/Linux.

## Resolution Rules

- Default settings resolve to English, LTR, non-pseudo catalog/profile.
- `en-*` resolves to English catalog and root `lang: "en"`.
- `ja-*` resolves to Japanese catalog and root `lang: "ja"`. The Japanese
  catalog may intentionally fall back to English strings until localization is
  staffed, but the selector still comes from Rust.
- `en-XA` resolves to accented pseudo locale with LTR layout.
- `ar-XB` resolves to bidi pseudo locale with RTL layout.
- Recognized RTL language families such as Arabic, Hebrew, Persian, Urdu, and
  related tags resolve to RTL under `TextDirectionPreference::Auto`, even when
  the current product catalog falls back to English.
- Explicit `TextDirectionPreference::Ltr` or `Rtl` overrides script defaults.
- Invalid, unsupported, or private-use-like language input falls back to a
  sanitized English profile. Resolved profiles and diagnostics must not echo
  raw locale input.

## Catalog And Formatting Rules

- User-visible product text uses stable catalog IDs from
  `apps/desktop/src/i18n/messages.ts` or its successor.
- Catalog templates use named placeholders such as `{roomName}`. Interpolation
  values are caller-owned data and must not be transformed by pseudo-localizer
  code.
- Pseudo-localization expands catalog-owned label text while preserving
  placeholders exactly. Tests must cover long labels, CJK samples, RTL samples,
  combining marks, and a bidi mode that is distinguishable from accented
  pseudo-localization.
- Core and adapters return structured codes, enum variants, and non-secret
  data. They do not return prose for product UI rendering.
- Remote/user text such as room names, display names, message bodies, snippets,
  attachment names, and reaction keys is rendered with `dir="auto"` at the UI
  boundary.
- Layout CSS uses logical properties for locale-sensitive spacing and
  positioning. Physical left/right assumptions are allowed only when the
  physical direction is intentionally part of the visual design and documented
  in the component.

## Headless Gates

Phase A gates:

- Rust unit tests for locale/display profile resolution, serialization,
  platform modifier labels, pseudo-locale tags, RTL/CJK handling, and
  private-data-free diagnostics.
- Catalog unit tests for ID coverage, pseudo expansion, placeholder
  preservation, RTL/CJK/combining samples, and interpolation behavior.

Phase B gates:

- Browser-headless tests apply the Rust-resolved root `lang`/`dir`, render
  pseudo/CJK/RTL/long labels without clipping or overlap, and verify
  `dir="auto"` on remote/user text.
- A raw user-facing string scan or reviewed allowlist is part of the UI gate
  before #4 can be closed.
