# Issues #253 and #254 Batch Design

Date: 2026-07-14

Issues:

- #253 — Verified session restore shows verification gate and repeats slow
  full-state preparation
- #254 — Formatted Markdown lists render HTML source whitespace as line breaks

## Scope and delivery

Implement both issues from `origin/main` in one isolated branch and publish one
non-squash pull request that closes both issues. The two changes are independent
at runtime, but share the same desktop release surface and verification pass.

This batch does not add new Matrix semantics, state DTOs, or a persisted
"verified" shortcut. It preserves Rust ownership of session admission and
formatted-message sanitization.

## #253: verified restore and sync ownership

### Problem

Restored sessions correctly consult the SDK crypto store for current-device
trust, but an authoritative `Verified` result currently starts an unconditional
unfiltered `full_state(true)` sync before projecting `Ready`. A restricted sync
may already be running, and another restricted lane is retained after the
normal `SyncActor` starts. These independent token owners make warm restore
slow, prevent offline admission, and permit overlapping classic-sync owners.

The frontend compounds this by rendering the verification gate while a `Ready`
session's normal sync is starting, reconnecting, or failed. During the actual
trust-read phase, the body says that trust is being checked while the heading
still says that the session must be verified.

### Security and state invariants

1. Current-device trust from the SDK crypto store remains authoritative on
   every restore. No application-level persisted `Ready` shortcut is allowed.
2. `Unknown` and `Unverified` remain fail-closed and may not project `Ready`.
3. A later authoritative downgrade from `Ready` still locks the session.
4. At most one classic-sync owner may be active for an account generation.
5. Product readiness is based on authoritative device trust, not successful
   network catch-up. Normal sync health is represented by normal shell state.
6. Stale generation messages and stale task completions remain ignored.

### Admission and handoff design

The provisional runtime subscribes to current-device trust before reading its
current value, retaining the existing no-lost-update ordering.

- If the initial authoritative value is `Verified`, do not start restricted
  sync. Persist the session as currently required, dispatch the authoritative
  trust transition, and wait for the normal reducer/projection acknowledgement.
- If the initial value is `Unknown` or `Unverified`, keep the session gated and
  start the restricted crypto/trust sync lane. This lane is owned only by the
  provisional account generation.
- If a later provisional trust update becomes `Verified`, first cancel and
  join the restricted lane. Only after that barrier may Rust dispatch the
  authoritative `Verified` transition.
- After the `Ready` projection acknowledgement, start the normal `SyncActor`.
  Do not start or retain a promoted restricted lane.
- Keep the trust subscription alive after promotion. SDK crypto-store updates
  driven by normal sync can therefore still produce an authoritative downgrade
  and lock the session.

The ownership sequence is consequently:

```text
provisional restricted owner (only when needed)
  -> cancel and join barrier
  -> Ready projection and acknowledgement
  -> normal SyncActor owner
```

A warm verified restore skips the first stage entirely. The unconditional
promotion full-state task, its detached completion message, its SDK helper, and
obsolete preparation diagnostics/tests are removed.

### UI design

- `CheckingTrust` uses the localized checking label as the main landmark label
  and heading. "Verify this session" is reserved for states that actually need
  user verification.
- A `Ready` session always renders the normal shell regardless of whether
  normal sync is starting, reconnecting, or failed.
- Existing shell sync status and restart controls handle those normal network
  states. The verification/preparation gate must not flash during them.

### Diagnostics

Extend the existing private-data-free startup/admission diagnostics so one
restore can be ordered through these phases:

1. restore/store open started and finished;
2. authoritative trust read finished with a coarse trust classification;
3. restricted catch-up started, stopped, or explicitly skipped;
4. `Ready` projection dispatched;
5. normal sync ownership started.

Diagnostics carry elapsed durations and generation-safe correlation only. They
must not include credentials, Matrix identifiers, message bodies, raw SDK
errors, or sync tokens.

### Test-first evidence

Add failing checks before implementation for:

- verified warm restore reaches `Ready` without releasing a full-state-sync
  override and without starting restricted sync;
- offline verified warm restore reaches `Ready` within a deterministic actor
  deadline, while normal sync may independently reconnect/fail;
- restricted-to-normal handoff never has two active owners;
- unknown and unverified restores never project `Ready`;
- later trust downgrade still locks a ready session;
- diagnostics are ordered, bounded, and private-data-free;
- checking-trust heading/landmark text is correct;
- ready sessions in starting, reconnecting, and failed sync states render the
  shell, including the existing restart action where applicable.

The local `gate_restore` headless scenario remains the integration proof. Its
bounded timing assertion is environment-specific and generous enough for CI;
the actor-level offline test provides the strict deterministic bound.

## #254: formatted-list source whitespace

### Problem

Rust already sanitizes Matrix formatted HTML and projects the safe structured
metadata. The React tokenizer preserves HTML text tokens, after which the
renderer currently converts every source newline in every text node into a
React `<br>`. Pretty-printed list markup therefore creates synthetic line
breaks, including invalid direct children under `ul` and `ol`. A grid-based
formatted-body root amplifies these anonymous rows.

### Rendering invariants

1. Ordinary HTML source whitespace follows browser whitespace collapsing and
   never creates a synthetic `<br>`.
2. An explicit sanitized `br` element creates exactly one line break.
3. Whitespace separating inline siblings is preserved so words do not join.
4. Whitespace-only direct children of `ul` and `ol` are omitted; their element
   children remain `li`.
5. `pre` and code-block content remains exact through the existing Rust-owned
   code-block projection.
6. Pretty-printed and minified equivalent HTML has equivalent semantics and
   layout, including nested lists and CJK content.
7. Link/highlight ranges keep their Rust-projected UTF-16 basis. Whitespace is
   not normalized before linkification in a way that shifts those ranges.

### Rendering design

Keep parsing, sanitization ownership, linkification, and code metadata
unchanged. At render time:

- render ordinary text directly instead of splitting on newline and inserting
  `<br>` nodes;
- contextually omit whitespace-only children directly under lists and at
  block-only formatting boundaries, while retaining meaningful inline spacing;
- retain the existing explicit `br` element renderer;
- use normal block/inline document flow for the formatted-body root instead of
  making each top-level token a grid item;
- express paragraph/list/nested-list spacing with adjacent block rules and
  compact list styles.

The shared `TimelineItemRow` formatted renderer supplies room, thread, and
focused-timeline bodies, so the fix applies uniformly. Plain reply previews do
not gain a new rich-HTML contract in this batch.

### Test-first evidence

Add failing component/style/browser checks for:

- pretty-printed paragraph/list/paragraph markup produces no synthetic `br`
  and list element children are only `li`;
- pretty and minified nested lists have equivalent DOM meaning and heights
  within a small browser tolerance;
- inline strong/emphasis siblings retain their separating space;
- an explicit `br` produces exactly one break;
- code/pre content remains exact;
- nested CJK lists remain compact and valid;
- the formatted root uses normal flow rather than grid layout.
