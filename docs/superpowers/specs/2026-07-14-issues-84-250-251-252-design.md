# Design: DM activity ordering and composer attachment parity (#84, #250-#252)

Status: approved for implementation
Date: 2026-07-14
Issues:

- https://github.com/shinaoka/koushi-matrix/issues/251
- https://github.com/shinaoka/koushi-matrix/issues/252
- https://github.com/shinaoka/koushi-matrix/issues/84
- https://github.com/shinaoka/koushi-matrix/issues/250

## Goal

Close the four issues in one durable branch and one non-squash PR. Issue #251
is an independent Rust-owned room-ordering batch. Issues #252, #84, and #250
form one ordered composer/media batch: establish destination-safe attachment
ingestion first, prepare image variants second, and then reuse both contracts in
the thread composer.

The completed product must not keep a second React-owned ordering algorithm,
must not retain attachment destination semantics only in a React `File` ref,
and must not make packaged desktop drag/drop behave differently from the
browser-headless contract.

## Upstream and existing-product reference

The existing main composer is the product reference for formatting, mentions,
emoji, shortcuts, paste, attachment selection, and scheduled-send exposure.
Matrix media captions remain one media event as already established by #34.
Matrix thread replies use the existing thread relation path; media and delayed
thread sends must carry the same root relation rather than emitting an
unthreaded room event.

Element-style desktop behavior accepts picker, paste, and drag/drop as
equivalent attachment entry points and prepares visible attachment state before
send. Koushi intentionally diverges from legacy Element Web image-plus-text
splitting: caption and media stay atomic in one event. Image preparation also
stays pre-send so Send uploads already-selected bytes and never opens a late
compression prompt.

## Batch 1: authoritative conversational activity ordering (#251)

### Data contract

`RoomSummary.latest_event` remains a preview/unread projection and is not
redefined as conversational activity. Replace the ambiguous use of
`last_activity_ms` with typed SDK-source data:

- an opaque SDK recency value, retained only where SDK recency ordering is
  explicitly required and never compared with epoch milliseconds;
- a Rust-owned optional conversational activity key carrying an epoch
  timestamp plus a coarse source kind such as message, encrypted message, or
  thread reply;
- no conversational key for membership, room creation, profile/state events,
  presence, typing, receipts, reactions, redactions, or replacements.

The SDK adapter classifies the latest relevant room activity before it reaches
state. An undecryptable encrypted room message still counts as conversation.
Edits and reactions may update the visible preview or target event but do not
advance the room as a new conversation. A thread reply advances its owning
room consistently with an ordinary message.

### Ownership and ordering

One comparator in `koushi-state` owns Active ordering and produces the sidebar
projection. It orders rooms with conversational activity by descending epoch
timestamp. Rooms without conversation follow them and use a documented stable
display sort key with room-id only as the final deterministic tiebreaker.
React renders projection order and removes its independent timestamp
comparator.

Restored sessions and incremental room-list updates feed the same typed keys
and comparator. Private-data-free diagnostics expose only the selected source
kind, presence/absence of conversation, and normalized ordinal relationships;
they do not print room IDs, labels, timestamps tied to real activity, or
message content.

## Batch 2: common attachment ingestion (#252)

### Composer target

Introduce a Rust-owned attachment/composer target:

- main room composer: room ID;
- thread composer: room ID plus root event ID.

Every stage, prepare, remove, select-variant, retry, cancel, and send command is
correlated to this target and a request/staged-item ID. Reducers reject stale
completions and unknown targets. Room selection, thread switching/closing,
logout, and account switching cancel or detach the old target so a late drop or
preparation result cannot appear in another composer.

### Ingestion adapter

Use one frontend adapter that accepts `File[]` from the file picker, clipboard,
or HTML5 drop and immediately submits the captured Rust target with file bytes
and metadata to the typed staging command. React may own drag-hover visuals and
the transient browser `File` read, but not staged lifecycle, destination,
ordering, retry, or send semantics.

Set Tauri WebView native drag/drop off and use the same HTML5
`DataTransfer.files` route in packaged builds. This avoids a second path that
would expose native local paths and makes browser and packaged behavior share
one adapter. The whole active composer surface accepts dragenter/dragover/drop,
shows a catalog-backed drop overlay, filters non-file items, and preserves the
`FileList` order.

Local paths never cross the boundary. Public `Debug`, diagnostics, QA output,
and issue evidence expose counts and coarse kinds only; filenames are visible
UI data but never diagnostic data.

## Batch 3: pre-send image preparation (#84)

### Rust-owned lifecycle and byte cache

Each staged item is one of `preparing`, `ready`, or `failed`. Serializable
`AppState` contains target, stable item/variant IDs, ordering, caption,
preparation status, selected variant, and display descriptors. Prepared bytes,
decoded pixels, and preview handles do not enter snapshots.

Core owns an account-scoped ephemeral prepared-media cache keyed by target,
item ID, and variant ID. Staging copies source bytes into this cache, starts
bounded preparation, and reliably dispatches correlated reducer actions.
Removing an item, changing target, logout, or account shutdown cancels work and
clears bytes. Send references an already-ready variant and never decodes,
resizes, re-encodes, or prompts.

### Image candidates

Move deterministic PNG/JPEG/WebP preparation into a portable Rust media
module used by core. Candidate generation follows #84:

- PNG: original PNG, resized PNG when useful, and WebP;
- JPEG: original JPEG, resized/recompressed JPEG when useful, and WebP;
- WebP: original WebP and resized/recompressed WebP;
- GIF, animated, or unsupported input: original only until an
  animation-preserving transform is designed.

Never select a candidate larger than the original by default. Preserve alpha
for PNG/WebP, strip EXIF/geolocation on re-encode, regenerate thumbnail data,
and report actual encoded MIME, extension, byte count, dimensions, and savings.
The settings mode supplies default selection policy; it does not create a
send-time Ask dialog. Failed preparation exposes Retry and Use original when
the original bytes are valid.

### Preview transport

The WebView requests preview bytes through an app-owned ephemeral media handle
and creates/revokes object URLs for display only. Preview URLs and prepared
bytes are not product state. Variant selection is a typed Rust command; the UI
updates only after the returned snapshot/delta selects that variant.

## Batch 4: thread composer parity (#250)

### Shared composer surface

Replace the separate minimal `ThreadComposer` behavior with shared composer
primitives. Surface capabilities are explicit data, while common controls and
interaction code provide bold, italic, link, list, code, mentions, emoji,
picker, paste, drop, shortcut, IME, and send behavior. Main and thread drafts
remain independently keyed.

Text formatting continues through the Rust markdown/formatted-body path.
Thread text sends use `SendReply`. Thread media sends use the selected prepared
variant and build one Matrix media event carrying the active thread relation.
Caption, cancellation, retry, local echo, and terminal settlement remain scoped
to the thread target.

Scheduled thread sends are supported rather than silently exposing a broken
main-only control. Extend the Rust scheduled-send target to room or thread and
preserve the thread relation through server-delayed and local fallback paths.
If a backend cannot represent the relation, Rust reports an unsupported
capability and the control is omitted from the capability projection; React
does not guess.

### Thread presentation cleanup

The thread timeline receives an explicit presentation context. In thread
context it suppresses `Start of conversation` and the root row reply-summary
pill without removing root content. The main room timeline retains the pill and
adds the Rust-projected latest-reply timestamp using the same adaptive local
formatter as timeline timestamps. A later reply updates the summary projection
and timestamp.

## State-machine and transport changes

The implementation must amend `docs/architecture/state-machine.md` before or
with reducer code for:

- typed conversational activity projection and authoritative room-list sort;
- target-scoped staged upload preparation, failure, retry, selection,
  cancellation, and terminal send transitions;
- thread media and scheduled-send targets;
- stale target/request guards and logout/account-switch cleanup.

Any new `AppState`, `CoreCommand`, `CoreEvent`, or delta field updates all
hand-maintained Tauri DTOs, TypeScript types, browser fakes, app/IPC harness
snapshots, generated wire artifacts, and serialization-contract tests in the
same change.

## Failure and privacy behavior

- Empty files and unsupported data are typed coarse failures.
- Transform failure never silently substitutes a different MIME type.
- Cache eviction or missing prepared bytes moves the correlated item to a
  retryable failure instead of uploading a different source.
- Late preparation/send results for a closed room/thread/account are ignored
  by reducer guards and their cache bytes are discarded.
- Debug output redacts filenames, captions, bytes, local paths, room/event IDs,
  transaction IDs, and raw SDK errors.
- QA uses synthetic filenames/content and token-only evidence.

## Verification strategy

Build each proof before its fix.

### #251

- Rust SDK/state fixtures: join-only DM versus recently messaged DM;
  undecryptable encrypted event; thread reply; ignored state/edit/reaction
  events; stable no-conversation fallback; restored/incremental equivalence.
- Browser test: rendered DM order is projection order and no React comparator
  remains.

### #252

- Browser tests dispatch real file drops on textarea, toolbar/body, and footer,
  plus paste and picker through the same adapter.
- Tauri configuration/contract tests prove packaged HTML5 file drop is enabled
  and native path payloads are not introduced.
- Platform packaged lanes cover an ordinary non-image file on macOS, Linux,
  and Windows; the available local Linux lane is required before PR, with
  macOS/Windows CI or documented platform gates required before merge.

### #84

- Pure Rust codec tests cover PNG alpha, resized PNG, JPEG recompression, WebP,
  real MIME/extension/size, metadata stripping, larger-output rejection,
  unsupported/animated original-only behavior, cancellation, stale results,
  and cache cleanup.
- Reducer/core tests cover preparing/ready/failed/retry/select/send and prove
  Send performs no conversion.
- Browser tests cover preview, candidate metadata, selection, error recovery,
  and disabled Send while preparing.

### #250

- Core/local homeserver tests prove formatted text, media caption, media thread
  relation, retry/cancel isolation, and scheduled thread relation.
- Browser tests cover all toolbar controls, mention/emoji, picker/paste/drop,
  IME/shortcuts, thread switching isolation, removed conversation chrome, and
  latest-reply timestamp updates only in the room timeline.

### Final gates

Run focused gates after every batch, then the repository-required state/core,
Tauri, TypeScript, browser-headless, IPC contract, secret scan, wasm, local
Conduit/Tuwunel, and packaged Linux GUI gates. Run an independent Codex diff
review against repository canon and this design, resolve verified findings,
then open one PR with `Closes #251`, `Closes #252`, `Closes #84`, and
`Closes #250` only after every acceptance item is proven. Merge with a merge
commit, never squash.

## Alternatives rejected

- Expanding #84 to own all drag/drop: hides a generic file-ingestion defect
  inside image compression and gives #250 no clean prerequisite.
- Keeping React `File` refs authoritative: cannot make target cleanup,
  preparation settlement, retries, or thread isolation Rust-owned.
- A separate native Tauri path adapter: duplicates ingestion and risks local
  path leakage when HTML5 drop can provide one cross-platform contract.
- Keeping the TypeScript Active comparator beside Rust ordering: preserves the
  drift that caused #251.
- Exposing scheduled send in threads without relation support: creates a
  successful-looking unthreaded send and violates product semantics.
